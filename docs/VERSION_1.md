# Dayo — Version 1 Documentation

*Last updated: 2026-04-16*

---

## 1. What Dayo Is

Dayo is a personal AI day planner built for people whose days don't look like a generic to-do list — homemakers, mothers, working moms, new mothers, and solo professionals. The core idea: a working mom with two kids and a homemaker recovering from C-section shouldn't see the same dashboard. Every user type gets a fundamentally different layout, a different AI prompt, a different set of sections — all anchored on one profile the user builds once in a conversational onboarding.

The product lives at `dayo-app.onrender.com` and runs as a mobile-first web app (max-width 430 px, designed to feel like a native iOS app).

---

## 2. Tech Stack

| Layer | Choice |
|---|---|
| Backend | Django 4.2 + Django REST Framework |
| Frontend | React 19 (Create React App) in `frontend/` |
| AI | LangChain + Google Gemini 2.5 Flash (all prompts, plan gen, chat, grocery, kids activities) |
| Database | SQLite in dev, PostgreSQL via `dj-database-url` in prod |
| Auth | Django session auth with CSRF tokens |
| Static | WhiteNoise for Django static file serving |
| Hosting | Render (web service + managed Postgres, auto-deploy on `main`) |
| PDF | ReportLab for kids activity PDFs |

**Why Gemini, not Claude:** cost at the generation frequency Dayo requires (every plan, every grocery list, every chat turn) was the deciding factor. Gemini 2.5 Flash hits the quality floor at a small fraction of Claude Sonnet's per-token price.

### 3.2 Three-layer dashboard

The dashboard is **config-driven**, not hardcoded per user type. Every section is rendered based on a JSON `custom_layout` array on `UserProfile`:

```json
[
  {"key": "schedule_alert", "visible": true, "locked": false},
  {"key": "meal_cards",     "visible": true, "locked": false},
  {"key": "grocery",        "visible": true, "locked": false},
  {"key": "kids_activities","visible": true, "locked": false, "added_by_ai": true},
  {"key": "custom_garden",  "visible": true, "locked": false,
                            "custom_label": "Garden care", "added_by_user": true}
]
```

Three layers feed into this array:

**Layer 1 — User-type defaults.** Hardcoded in `planner/section_registry.py::DEFAULT_LAYOUTS`. Pick "parent", get meals + grocery + kids activities + housework + me-time + notes. Pick "professional", get deep work + priorities + meetings + compact meals. This runs once at the end of onboarding via `build_initial_layout()`.

**Layer 2 — AI extraction.** During onboarding, the `ProfileBuilderAgent` extracts a `planning_modules` list (e.g. `["grocery", "exercise", "kids_activities", "selfcare"]`) from natural conversation. `MODULE_TO_SECTION` maps each module to a section key and appends it (marked `added_by_ai: true`) if the user-type default didn't already include it.

**Layer 3 — User customisation.** On the OnboardingPreview screen and later via `/settings/dashboard` (CustomiseDashboard), the user can add, remove, rename, and reorder sections. A user can also add up to 3 **custom sections** — fully named by them, tracked with a `custom_label`, stored in `CustomSectionList`/`CustomSectionTask` tables, and generated by the same AI pipeline as built-in sections.

The frontend `DynamicDashboard.jsx` reads `profileData.custom_layout` and renders sections in order. If layout is empty for some reason, it falls back to `FALLBACK_LAYOUTS` keyed on `userType`.

### 3.3 User types (5)

| Type | Target | Dashboard flavour |
|---|---|---|
| `parent` | Homemaker with kids | Meals, kids activities, grocery, housework, me-time |
| `new_mom` | Mom with infant | Baby schedule, essentials checklist, recovery exercise, mom meals, milestones |
| `working_mom` | Working mother | Schedule alerts, priorities, kids activities, evening routine |
| `homemaker` | Homemaker without kids | Meals, grocery, housework, me-time |
| `professional` | Working professional | Deep work, priorities, meetings, compact meals |

A `student` type existed earlier and was removed; some leftover references still exist in `ai_context.py` and `ScheduleEvent.EventType` (see §7).

### 3.4 AI integration

All AI calls go through LangChain's `ChatGoogleGenerativeAI` wrapper. Each service instantiates its own LLM with its own temperature (plan gen: 0.7, grocery: 0.5, chat: 0.5 with tools bound). System prompts are assembled dynamically per user by `AIContextAssembler`, which pulls:

- Base profile (name, user type, wake/sleep, location, family size, home help)
- Preferences (dietary, cuisine, meal types, favourites)
- Schedule events for today + next 7 days
- Children's info (with age calculation)
- New-mom context (weeks postpartum, C-section, support type, breastfeeding)
- Today's DayPlan, grocery list, housework (when relevant)
- Swap patterns (rejected meals) and deletion logs (removed housework tasks) for learning

Each user type gets a **different JSON template** fed to Gemini (see `JSON_TEMPLATES` in `plan_generator.py`). Professional gets `deep_work`, `priorities`, `meetings`, compact meals. New mom gets `mom_meals`, `baby_schedule`, `milestones`, `essentials_check`. This is why plans feel qualitatively different across types, not just different sections.

### 3.5 Authentication

Django session auth. Flow:
1. Frontend calls `GET /csrf/` on first load → sets `csrftoken` cookie
2. User registers or logs in → `sessionid` cookie set
3. All state-changing requests include `X-CSRFToken` header (read from cookie)
4. `credentials: 'include'` on every fetch so cookies ride along

`RegisterView` explicitly sets `authentication_classes = []` to skip CSRF on the register endpoint (otherwise first-ever request has no token).

### 3.6 Hosting & deploy

Render auto-deploys on push to `main`. Django serves the React build from `staticfiles/` via WhiteNoise. Backend and frontend share one origin in production (`/api/v1/*` on the same host), but in dev the frontend runs on `:3000` and talks to Django on `:8000` (hence `API_BASE` branches on `process.env.NODE_ENV`).

## 4. Feature Walkthrough

### 4.1 Onboarding

The longest user flow — 4 screens — because getting the profile right determines everything downstream.

**`OnboardingPage`** — A traditional form: name, user type (5 options), city, wake time, sleep time. No AI. Just collects the basics so the chat agent has a starting point.

**`OnboardingChat`** — The AI conversation. `ProfileBuilderAgent` runs a 3-5 turn Gemini conversation that feels natural but is secretly extracting a structured profile. The agent has a hidden checklist — dietary restrictions, cuisine preferences, family size, kids, schedule patterns, health conditions, exercise interest, self-care desires — and steers the conversation to cover the ones it hasn't filled in yet. Each AI response optionally comes with **chip buttons** (quick reactions the user can tap instead of typing). After enough data is collected, the agent emits `is_complete: true` along with a `profile_data` dict, a `confidence` score, and `section_reasons` (why each section was suggested).

**`OnboardingPreview`** — The user sees what the AI extracted: their suggested sections, reasoning behind each, and the option to add/remove/reorder before committing. They can also add up to 3 custom sections. On confirm, `POST /profile/layout/` persists the layout and `POST /plans/generate/` kicks off the first day plan.

**`OnboardingLoading`** — Dark-themed fullscreen loader shown while Gemini generates the first day plan (5-15 seconds typically).

**Key data collected end-to-end:**
- OnboardingPage: `display_name`, `user_type`, `location_city`, `wake_time`, `sleep_time`
- OnboardingChat: `dietary_restrictions`, `cuisine_preferences`, `health_conditions`, `family_size`, `home_help_type`, `children[]`, `schedule_events[]`, `planning_modules`, `module_preferences`, new-mom specifics (`baby_name`, `baby_date_of_birth`, `is_breastfeeding`, `had_csection`, `support_type`)
- OnboardingPreview: final `custom_layout`

### 4.2 Plan generation

One user action (tap "Plan my day") triggers a cascade:

1. `PlanGenerator.ensure_meals_ahead(profile)` — if fewer than 3 days of meals are already planned for this week, generates the next block via `generate_weekly_meals` (a single Gemini call that returns 7 days of breakfast/lunch/dinner).
2. `PlanGenerator.generate_day_plan(profile, target_date)` — generates today's non-meal sections (housework, priorities, kids activities, notes, quick chips) using the user-type-specific JSON template. If weekly meals already exist, they're preserved.
3. If the user has children → `KidsActivityGenerator.generate_weekly_plan(profile)` — generates Mon-Fri activities per child (story + worksheet + coloring).
4. If no grocery list exists for this week → `GroceryGenerator.generate_grocery_list(profile)` — extracts ingredients from the week's meals, consolidates, scales for family size.

All four run synchronously within the request. The frontend sees a single loading state.

**Failure modes are now logged** (post-2026-04-16 fix) — previously, any of these could silently fail without explanation.

### 4.3 Meals

Meals are generated **weekly**, not daily, to maintain variety. `generate_weekly_meals` prompts Gemini for a 7-day block with:

- Every meal nutritionally balanced (protein + carbs + healthy fats + fibre)
- Specific named dishes (e.g. "Rajma Chawal with Green Salad", not "healthy lunch")
- English names for dishes (no "Mercimek Çorbası" — write "Red Lentil Soup")
- Health-condition-aware (PCOS → low-GI options; diabetes → balanced carbs; etc.)
- Varied — avoids repeating last week's meals (fed in as context)
- Family-size scaled

Users can swap a meal, substitute it with a specific request ("something lighter"), or change the whole day's meals via three dedicated endpoints. Rejections are logged to `MealSwapLog` and fed back into future generations — the AI learns what this user doesn't like.

Users can favourite meals via the pin icon. Favourites are surfaced in future weeks' generation as "rotate some of these in".

### 4.4 Grocery

One `GroceryList` per week. Items grouped by category (produce, dairy, grains, protein, spices, snacks, other). Key behaviours:

- **Auto-generated** from the week's meals when the plan is generated, or lazily when the dashboard loads if none exists (self-healing added 2026-04-16).
- **Pantry-aware** — items in `UserPantryItem` are excluded.
- **Essentials-aware** — items unchecked in `EssentialsCheck` for 2+ consecutive days get added automatically.
- **Carryover** — when a new week's list is generated, unchecked user-added items from the previous list carry over.
- **Mid-week additions** — user can add any item to the active list via a quick-add input.
- **"Done shopping"** — marks the list completed. Next week's list starts fresh.
- **Share** — generates a formatted text version for WhatsApp/Messages.

The "Grocery sorted" empty state (which used to confuse users) now auto-heals on dashboard load.

### 4.5 Housework

One `HouseworkList` per day. Two sources:

1. **`HouseworkTemplate`** — recurring tasks the user configured (e.g. "Clean bathroom, Mon+Thu"). Always included on matching weekdays.
2. **AI-generated** — Gemini fills in the rest up to a target count that varies by user type:
   - Homemaker/parent: 6-8 tasks
   - Working mom: 4-6 tasks
   - New mom: 1-2 tasks (heavily constrained — see postpartum rules below)

When a user deletes an AI-generated task, it's logged to `HouseworkTaskDeletionLog` so the AI learns not to suggest it again.

**New-mom housework is special** (2026-04-16): the prompt now branches by weeks postpartum:
- Weeks 0-2: empty array (recovery only)
- Weeks 2-6 (or <12 if C-section): 1 micro-task under 5 minutes
- Weeks 6+: 1-2 light tasks under 10 minutes
- If partial domestic help: heavy tasks allowed, framed as helper instructions

With explicit forbidden list (no vacuuming, mopping, scrubbing, carrying laundry, etc.) and a safe-example list (fold baby clothes, rinse bottles, wipe counter).

### 4.6 Kids activities

Weekly plan (Mon-Fri) generated per child. Each day has:
- A short illustrated **story** (title + emoji + illustration description + body text)
- A **worksheet** with 3 age-appropriate activities (from 20 types: tracing, counting, mazes, matching, dot-to-dot, word search, math, crossword, riddles, etc.)
- A weekly **theme** shared across all children

Mon is unlocked immediately. Tue unlocks when Mon is marked read/downloaded. And so on — a soft gating mechanism to pace engagement.

Users can download any day as a **PDF** (generated via ReportLab, `kids_activity_pdf.py`). When all days for a child are complete, the system **auto-regenerates** a fresh week for that child.

If a user adds a child manually via Profile (added 2026-04-16), `kids_activities` is auto-added to their layout. The section also now shows a visible "Try again" card if generation fails instead of silently rendering nothing.

### 4.7 Chat

Conversational AI interface over the user's plan. Backed by Gemini with **tool calling**:

- `add_grocery_item`, `remove_grocery_item`
- `swap_meal`, `add_housework_task`, `mark_housework_done`
- `cancel_schedule_event`, `add_schedule_event`
- `add_metime_activity`, `add_errand`

When the user says "add milk to grocery", the AI returns a message **plus** a `pending_action` JSON blob. The frontend shows a confirmation card ("Add milk to your grocery list?"). Only on user confirm does the DB write execute (`execute_tool()`). Users can cancel. This two-step prevents accidental mutations from AI hallucinations.

Chat history is persisted in `ChatConversation` + `ChatMessage`. Multiple conversations supported.

### 4.8 Schedule

Calendar of `ScheduleEvent` rows — appointments, school pickups, kids' classes, meetings, meal prep windows. Each event has:
- `event_type` (11 choices)
- Optional link to a `Child`
- Recurrence pattern (daily, weekdays, weekly, specific days)
- Travel time (minutes) — used for "leave by" calculations
- Active flag

Events feed into the AI context so plans and reminders respect them. A user's "leave by 16:30 for Arya's football" alert is driven by this.

### 4.9 Custom sections

Users can add up to 3 fully custom sections (e.g. "Garden care", "Elderly parent meds", "Side business"). The system:
- Adds the key to `custom_layout` with `added_by_user: true` and a `custom_label`
- Generates daily task content via the same AI pipeline (custom system prompt fragment in `plan_generator._build_custom_sections_prompt`)
- Stores the tasks in `CustomSectionList` + `CustomSectionTask`
- Renders via the generic `CustomSection.jsx` component
- Tracks deletions in `CustomSectionDeletionLog` for AI learning

### 4.10 Profile & customise dashboard

**ProfilePage** — standard CRUD on `UserProfile` + `Child` + favourites + pantry. Nothing fancy.

**CustomiseDashboard** — drag to reorder, toggle visibility, remove sections, add sections from the registry, add custom sections. Saves `custom_layout` via `PATCH /profile/layout/`.

         │                 ├── CustomSectionDeletionLog

The `custom_layout` JSONField on `UserProfile` is the key integration point — it's the only thing that connects model-level identities (section keys) to render-level behaviour.

`DayPlan.plan_data` is a free-form JSONField whose structure depends on `user_type`. This trades strict schemas for flexibility: a new mom's plan has `baby_schedule`, a professional's has `deep_work`, and both are valid under the same model.

## 6. Journey — How We Got Here

This section reflects the user-visible evolution of the project, inferred from git history, CLAUDE.md notes, and the recent session's fixes.

### 6.1 Starting point

The initial commit `6143858 — "Dayo v1 — initial release"` shipped a functional core: Django + React, auth, onboarding, plan generation, grocery, housework, chat. From there, a sequence of fixes addressed production pain points:

- `0f409c7` — Disable CI mode in React build (warnings were blocking deploys)
- `d5a0a5d` — Add missing ReportLab dependency (kids PDFs crashed in prod)
- `7d8e17b` — Fix hardcoded localhost in OnboardingChat (production calls were failing)

### 6.2 The shift from HomemakerDashboard to DynamicDashboard

The first dashboard was hardcoded for homemakers — `HomemakerDashboard.jsx` still sits in the repo, unused. When the product expanded to 5 user types, this didn't scale. The solution was the 3-layer dashboard with `custom_layout` as the data model — one renderer (`DynamicDashboard`), one registry (`section_registry.py`), five default layouts, infinite user-level customisation. The legacy file was left in the repo but orphaned.

### 6.3 Removing the `student` user type

Early versions had a `student` type with academic sections (assignments, exams, study blocks). The type was removed, but residual code remains: `ai_context.py` still checks for it, and `ScheduleEvent.EventType` still has `CLASS`, `STUDY`, `EXAM`, `ASSIGNMENT_DUE` choices. These are effectively dead code now.

### 6.4 The fix wave (this session, 2026-04-15 to 2026-04-16)

A concentrated round of production bug fixes deployed as four commits:

**`83cdfee` — Grocery + kids silent failures.** Fixed three issues:
- Grocery generator used to create the empty `GroceryList` row **before** calling the AI, so a Gemini failure left a zombie empty list that blocked all future auto-gen. Refactored to do AI first.
- `GroceryCurrentView` now **self-heals**: if no active list exists but meals are planned, it auto-generates on dashboard load. No more stuck "Grocery sorted" state.
- Adding a child via `ChildViewSet.perform_create` now auto-appends `kids_activities` to `custom_layout`. Previously, homemaker/new-mom/professional users who added a kid later never saw the section.
- `KidsActivitiesSection` replaced silent `return null` on failure with a visible retry card + Try Again button that hits `/kids-activities/generate/` (which now wipes empty-shell plans and regenerates).
- Multiple `except Exception: pass` blocks replaced with `logger.error` so Render logs surface failure reasons.

**`6d5eb17` — Postpartum-safe housework.** Added `_new_mom_housework_rules()` which branches the housework AI prompt by weeks postpartum and C-section status. Explicit forbidden list (vacuum/mop/scrub/carry laundry baskets) and safe-example list (fold, wipe, rinse, tidy). Special case for partial domestic help (heavy tasks OK, framed as helper instructions).

**`cb32a4c` — Mobile UI.** Fixed four mobile problems:
- **iOS zoom on input focus** — bumped `.auth-input` to 16px and added a media-scoped rule (`!important`) forcing all inputs to 16px on phones. Covers inline-styled inputs in grocery/profile/chat too.
- **100vh clipping on mobile Safari** — replaced `100vh` with `100dvh` (with `100vh` fallback) in 6 places (CSS + 3 inline JS spots).
- **Home-indicator overlap** — added `env(safe-area-inset-bottom)` to `.bottom-nav` padding and the `.app-shell` bottom padding.
- **Nav width misalignment on larger phones** — bumped `max-width` from 390 px to 430 px on `.app-shell` and `.bottom-nav` (was mismatched with other files that used 430 px).
- Added `viewport-fit=cover` + `apple-mobile-web-app-capable` meta tags.

**`d7fc725` — Onboarding chat send button.** The send button was `onClick={sendMessage}` where `sendMessage(directText)` expected an optional string. React passed the click event as `directText`, `.trim()` threw, the button silently did nothing. Enter on the keyboard worked because its handler called `sendMessage()` with no args. Wrapped in `() => sendMessage()`.

### 6.5 Lessons encoded in the code

Reading the codebase, you can see **specific opinions** about users baked into prompts:

- "NEVER use language like: lose weight, get your body back, post-baby diet" (new mom meals)
- "Frame as: recovery, energy, strength for carrying baby" (new mom exercise)
- "Frame as medically necessary, NOT a luxury" (new mom me-time)
- "Example: 'Today's meals are gentle on your PCOS — low GI, anti-inflammatory'" (health-aware meals)
- "No leaving the room" (flying-solo new mom)

These aren't incidental. Each of these lines exists because the default LLM output would otherwise produce something tone-deaf for the target user. V1 is as much a collection of carefully-tuned system prompts as it is Django + React code.

## 7. Known Limitations & Tech Debt

Categorised by how urgent they are.

### 7.1 Must fix before a meaningful user base grows

- **In-memory onboarding sessions** — `_onboarding_sessions = {}` in `views.py` is a module-level dict. Scales to exactly one server instance. Move to DB or Redis.
- **No rate limiting** on plan/grocery/kids-activity generation. A user (or bot) can spam `/plans/generate/` and burn through your Gemini quota.
- **No timeouts on Gemini calls.** A stuck upstream means a frontend request hangs until the browser gives up. Set an explicit timeout on the LangChain client.
- **Chat history has no pagination.** Loading a long conversation pulls every message.
- **`DayPlan.plan_data` is unvalidated JSON.** Garbage from a bad AI response (or a schema change) can render the dashboard blank with no error.

### 7.2 Should fix for user trust

- **Section registry duplication.** `planner/section_registry.py` (backend) and `DynamicDashboard.jsx::SECTIONS` (frontend) both list sections. Adding a new section requires updating both in sync. Generate one from the other at build time, or fetch the registry at runtime.
- **Meal endpoints duplicate logic** — `SwapMealView`, `SubstituteMealView`, `ChangeMealView` overlap heavily. Consolidate.
- **Dead `student` user type code** in `ai_context.py` and `ScheduleEvent.EventType`. Remove it.
- **Orphaned `HomemakerDashboard.jsx`.** Delete.
- **No timezone handling on reminders.** They're stored in UTC, shown without conversion. A user in IST sees UTC times.
- **No accessibility** — no ARIA labels, no keyboard-focus styles, colour-only alerts (amber banners etc.).

### 7.3 Nice-to-have polish

- No tablet breakpoints. Fine, but a missed opportunity.
- No offline mode / service worker. No PWA install prompt.
- No real-time sync across tabs/devices (WebSocket or polling).
- No section-level lazy loading — every dashboard section mounts even if scrolled past.
- No tests. Zero `tests.py` content. Smoke tests over the critical generators would catch a lot.
- Kids activity PDFs are regenerated every download (not cached on disk).

### 7.4 Data model smells

- `UserProfile` has ~30 fields, several nullable, several that only apply to one user type (`baby_name`, `had_csection`, `support_type` only matter for `new_mom`). Could split into a `NewMomProfile` OneToOne, but arguably the simplicity wins.
- Deletion logs (`HouseworkTaskDeletionLog`, `CustomSectionDeletionLog`) grow forever. No prune job.

## 8. V2 — What to Build Next

A prioritised roadmap. I've grouped by **user value** (does this make the app materially better for the target user) vs **engineering** (does this make the codebase sustainable).

### 8.1 High user value — these change how people feel about the app

**1. Mobile app (PWA or React Native)**
Right now it's a website that looks mobile-ish. Make it installable, push-notification-capable, offline-viewable for today's plan. Start with PWA (service worker, manifest, push) to avoid a full rewrite. React Native if PWA hits limits.

**2. Push notifications**
Leave-by times for school pickups. Meal prep starts. Baby feeding reminders. End-of-day wind-downs. Right now `Reminder` model exists but nothing sends them. Build the sender job (cron or Celery), wire up web push + FCM/APNS.

**3. Voice input**
"Add milk to grocery" is faster spoken than typed — especially for new moms holding a baby, or homemakers cooking. Browser Web Speech API is free; start there. Integrate into the chat input and possibly the quick-chip area on the dashboard.

**4. Photo-based grocery check-in**
Snap a photo of what's in the fridge → AI reads it → pantry items auto-updated, grocery list adjusted. Gemini is multimodal; this is a feasible prompt pattern. Huge friction reduction vs. manually maintaining `UserPantryItem`.

**5. Partner / family sharing**
Two parents, one shared dashboard (or their own views sharing the grocery + schedule). Mom schedules Arya's football, dad gets the leave-by alert. This is a data-model change (scope everything to a `Household` one level up from `UserProfile`) so the sooner the better.

**6. Meal learning → auto-tuning**
Right now `MealSwapLog` captures rejections, but the feedback loop is coarse. V2: surface "You've swapped out rajma 3 times — want to exclude it?" Actively ask rather than quietly re-prompt.

**7. Deeper health integration**
- Period tracking → PCOS-aware meal adjustments across the cycle (iron-rich during menses, etc.)
- Breastfeeding tracking → calorie adjustments
- Sleep tracking integration → reschedule heavy tasks on bad-sleep days
These already feel like natural extensions of existing user-type awareness.

**8. Shopping integration**
One-tap order the grocery list via BigBasket / Blinkit / Instacart. The hardest part is not the API integration — it's keeping item names mapped across catalogs.

### 8.2 Medium user value — polishes that compound

**9. Section templates / presets**
"I'm a new mom with a C-section and no help" → one tap applies a carefully tuned template. New-mom isn't a monolith; a menu of templates under the user type would reduce onboarding friction.

**10. Streaks / gentle accountability**
Not gamification for its own sake — gentle. "You've done your me-time 5 days this week" as a quiet footnote, not a badge.

**11. Week view**
Right now the dashboard is today-only. A weekly view for meal planning, grocery prep, kids activities. The data exists (`WeeklyMealsView` returns it); the UI is missing.

**12. Dark mode**
Low effort given the clean design system. High appreciation from users in bed with a baby.

**13. Better empty states**
The Grocery empty state caused a real bug today. Audit every "no data yet" UI and make sure it's either (a) immediately actionable or (b) clearly informational.

### 8.3 Engineering / foundation — these unblock everything else

**14. Tests**
Start with service-level tests on the four generators (grocery, housework, kids, plan). These are the highest-failure surfaces. Mock Gemini, assert DB writes match prompt outputs. Then integration tests over the onboarding flow.

**15. Background jobs**
Celery or Django-Q. Move plan generation, grocery generation, and kids activity generation off the request thread. Gemini calls can take 10+ seconds; blocking a request for that long is a bad mobile experience. Return immediately with a "generating" status and WebSocket the result when ready.

**16. Observability**
Add Sentry (or similar). Right now `logger.error` goes to Render logs — usable but not searchable. Set up alerts on Gemini failure rate, 500 rate, onboarding-completion rate.

**17. DB migration to Postgres in dev too**
SQLite in dev + Postgres in prod means bugs that only show up in prod (JSON field behaviour differs). Move dev to Postgres via Docker Compose.

**18. Typed schemas**
`DayPlan.plan_data` being free-form was a reasonable v1 choice, but it's now the biggest source of "the dashboard rendered blank" bugs. Define Pydantic models per user type, validate on save, fail loud with a specific error.

**19. Consolidate meal endpoints**
Swap + substitute + change are three endpoints with overlapping logic. One endpoint, a `mode` parameter, shared validation.

**20. Single section registry**
Make `section_registry.py` the source of truth. Expose it via `/sections/` (already exists). Have the frontend fetch it on app load and build its component map dynamically instead of hardcoding `SECTIONS` in `DynamicDashboard.jsx`.

### 8.4 Things worth explicitly **not** doing

- **Don't add gamification** (badges, XP, streaks with punishment for breaks). This demographic will churn.
- **Don't add social features.** No followers, no likes. The app's value is private.
- **Don't over-generalise.** "Any user type" sounds good but the product's soul is that it's specific. If you add a sixth user type, make it as specifically tuned as the first five.
- **Don't premature-optimise scale.** A React Native app, multi-region deploys, etc. before you have 10k users is distraction.
